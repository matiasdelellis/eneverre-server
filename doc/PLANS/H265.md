# H265 (HEVC) support ‚Äî implementation plan

Status: **not implemented.** The embedded media engine handles **H264 video +
AAC/G711 audio** only. A camera that sends H265 is detected and reported (see
[Diagnostics](#diagnostics)) but not recorded, relayed or shown. This document
scopes what adding H265 would take.

## TL;DR

- **Record + RTSP relay + playback (download/HLS-VOD): a modest, well-bounded
  addition.** All the hard media code already exists in the libraries we use;
  the work is wiring a parallel branch to H264 in the recorder. Estimate
  ~150‚Äì250 lines + tests.
- **Web live view (MSE): blocked by the browser, not by us.** Browsers can't
  reliably decode HEVC in MediaSource (Safari yes, Chrome only with hardware
  HEVC on some platforms, Firefox no). Making web live work for H265 on every
  browser needs **transcoding H265‚ÜíH264**, which is CPU-heavy and breaks the
  engine's zero-re-encode design ‚Äî a separate, much larger effort.

## What already exists (no media algorithms to write)

| Piece | Location |
|---|---|
| RTSP `format.H265` (VPS/SPS/PPS) + `rtph265` RTP decoder | `gortsplib/v5/pkg/format` |
| H265 DTS extractor | `mediacommon/v2/pkg/codecs/h265.DTSExtractor` |
| `h265.IsRandomAccess`, `NALUType_VPS/SPS/PPS/IDR_*` | `mediacommon/v2/pkg/codecs/h265` |
| fMP4 codec `mcodecs.H265{VPS,SPS,PPS}` (+ `IsVideo()`) | `mediacommon/v2/pkg/formats/mp4/codecs` |
| `fmp4.Sample.FillH265(ptsOffset, au)` (AVCC + sync flag) | `mediacommon/v2/pkg/formats/fmp4` |
| `fmp4.Init.Marshal` writes the `hvcC` box automatically | `mediacommon/v2/pkg/formats/fmp4` |

Reference implementation to mirror: upstream MediaMTX's fMP4 recorder handles
H265 the same way (this engine is a port of it).

## Component breakdown

### 1. Playback (download + HLS VOD) ‚Äî free
`internal/media/playback` is **codec-agnostic** (no H264/H265-specific code): it
reads the segment's init and re-muxes. Once segments are H265 fMP4, playback
works. Browser playability of the resulting clip still depends on client HEVC
support, but the server side is done.

### 2. RTSP relay (Android live) ‚Äî free once the recorder accepts H265
`liverelay` is a raw RTP passthrough ‚Äî codec-independent. Today the recorder
returns before publishing the source when there's no H264 track, so the relay
never gets packets. With H265 wired into the recorder, `OnSource`/`OnRTP` fire
and the relay serves H265 natively (Android `RtspMediaSource` decodes HEVC).

### 3. Recorder ‚Äî the real work (`internal/media/recorder/recorder.go`)

Contained, mechanical changes mirroring the H264 path:

- **Generalize two H264-typed fields** (l.82‚Äì83): `videoCodec *mcodecs.H264` and
  `dtsExtractor *h264.DTSExtractor`. Introduce a small interface for the DTS
  extractor (`Initialize()` + `Extract(au, pts) (int64, error)` ‚Äî both h264 and
  h265 extractors satisfy it) and hold the codec params per-codec.
- **Video-track setup** (l.158‚Äì170): try `format.H264`; if absent, try
  `format.H265`. For H265: `addTrack(mcodecs.H265{VPS,SPS,PPS})`, create the
  `rtph265` decoder, `Setup`, and wire `OnPacketRTP` ‚Üí new `processH265`.
- **`processH265`** ‚Äî parallel to `processH264` (l.365):
  - NALU type = `(nalu[0] >> 1) & 0x3F`.
  - Capture **VPS(32)/SPS(33)/PPS(34)** (H265 adds VPS vs H264's SPS/PPS).
  - VCL detection: H265 VCL types are `0..31`.
  - Inject VPS+SPS+PPS into random-access AUs (RTSP delivers them out-of-band in
    the SDP), same rationale as the H264 SPS/PPS injection.
  - Use the h265 DTS extractor; on the first random-access AU, start.
  - `sample.FillH265(int32(pts-dts), au)` (sets payload + `IsNonSyncSample` via
    `h265.IsRandomAccess`).
- **Relax the hard "H264 only" gate**: `ErrNoSupportedVideo` becomes "no H264 or
  H265"; the diagnostic message stays.

Everything downstream is unchanged: segment rotation keys off
`Codec.IsVideo()` (true for H265) and `IsNonSyncSample`; the `mtxi` box, index,
retention and playback are codec-agnostic.

### 4. Web live (MSE) ‚Äî the wall

Even fully wired (`case *mcodecs.H265` in `live.SetTracks` + an `hvc1.*` codec
string built from the H265 SPS), the **browser** must decode HEVC in MediaSource:

| Browser | MSE HEVC |
|---|---|
| Safari | yes (`hvc1`) |
| Chrome/Edge | only with hardware HEVC + platform support; often off |
| Firefox | no |

Options for H265 web live, in order of cost:
1. **Do nothing** ‚Äî web live unavailable (or Safari-only) for H265 cameras;
   fall back to a snapshot/thumbnail in the wall, keep RTSP for Android. Cheap.
2. **Transcode H265‚ÜíH264** for the MSE broadcaster (and optionally recording).
   Needs an H265 decoder + H264 encoder (ffmpeg/cgo, or a Go pipeline),
   CPU-heavy per camera, breaks the "no re-encode" design. Large, separate
   effort. Only justified if H265 web live on all browsers is a hard
   requirement.

## Effort & risk

- Components 1‚Äì3 (record/relay/playback): **~150‚Äì250 lines, low risk** ‚Äî all
  library APIs exist, mirrors upstream MediaMTX. A camera with both H264 and
  H265 is unusual; cameras send one video codec.
- Component 4 (universal web live): **large**, and revisits a core design
  principle. Recommend deferring unless required.

## Test plan

Reuse the local RTSP test rig with an H265 pattern:

```bash
# publish H265 to the example server
ffmpeg -re -f lavfi -i "testsrc=size=320x240:rate=15" \
  -c:v libx265 -x265-params keyint=15:min-keyint=15 -pix_fmt yuv420p \
  -f rtsp rtsp://localhost:8560/h265

# point a camera's `source` at it, then verify:
#  - segments appear + index grows (recording)
#  - ffprobe the RTSP relay shows codec_name=hevc (relay)
#  - playback/get returns a playable HEVC clip (playback)
```

## Diagnostics

Already implemented: when a camera offers no supported video codec, the recorder
returns `recorder.ErrNoSupportedVideo` wrapping the offered codec list, and the
engine logs it once and backs off to 30s instead of reconnecting every second:

```
level=WARN msg="media/recorder: camera codec not supported (recording/live disabled for it)"
  camera=jardin err="no supported video codec (recorder handles H264 only); stream offers: H265"
```

## Recommendation

Implement components 1‚Äì3 (record + relay + playback) if/when an H265 camera is
in play ‚Äî it makes such a camera fully usable for recording, Android live and
playback download. Treat universal H265 web live (component 4) as out of scope
unless transcoding is explicitly wanted; document it as a browser limitation.
